## 1. What is Java?

Java is an **object-oriented programming language** known for its **"Write Once, Run Anywhere"** capability — code written in Java can run on any platform that has a JVM installed. This platform independence is one of Java's biggest strengths.

> **Interview line:** Java is a strongly-typed, compiled-then-interpreted, platform-independent OOP language. Source code compiles to bytecode, which the JVM interprets/JIT-compiles to native machine code at runtime.

- **Strongly Typed** → Every variable has a fixed data type; type errors are caught at compile time.
- **Compiled** → `javac` converts `.java` source code into platform-independent **bytecode (****`.class`****)**.
- **Interpreted** → The **JVM** loads and executes the bytecode.
- **JIT Compiled** → Frequently executed (“hot”) bytecode is compiled into native machine code for faster execution.
- **Platform Independent** → The same bytecode runs on any OS that has a compatible JVM (“Write Once, Run Anywhere”).
- **Object-Oriented (OOP)** → Programs are built using classes and objects, supporting encapsulation, inheritance, polymorphism, and abstraction.
### Java Overview

- **Problems Before Java** → Programs were platform-dependent, less secure, and difficult to maintain across different operating systems.
- **Why Java Was Invented** → To build **portable, secure, and object-oriented** applications that could run on any device with a JVM.
- **History** → Developed by **James Gosling** at **Sun Microsystems** in **1991** as **Oak**; renamed to **Java** in **1995** due to trademark issues.
- **Evolution** → Grew from **Java 1.0** to **Java 21**, adding features like Generics, Streams, Lambdas, Modules, Records, and Virtual Threads.
- **LTS & Key Features** → Major **LTS (Long-Term Support)** releases: **Java 8, 11, 17, 21**. Core features: Platform-independent, Strongly Typed, OOP, Secure, Robust, Multithreaded, Automatic Garbage Collection, High Performance (JIT).

## 2. JDK · JRE · JVM

### The Nesting Relationship
```
JDK
├── Development Tools (javac, javadoc, debugger, etc.)
├── Source Code (core class source files)
└── JRE
    ├── JVM
    ├── Core Classes (compiled — String, Integer, ArrayList, etc.)

```

### JDK — Java Development Kit
- Comprehensive environment to **develop** Java applications.
- Contains: `javac` compiler, `javadoc`, debugger, source files for core classes, and the JRE.
- **Installing JDK = automatically installing JRE + JVM.**

### JRE — Java Runtime Environment
- Everything needed to **run** (not develop) Java apps.
- Contains: JVM + compiled core classes + configuration files.
- Has **no development tools** — meant for end users.
- JRE can be installed **without** JDK.

### JVM — Java Virtual Machine
- The actual **engine** that executes bytecode.
- Platform-dependent (different JVM for Windows/Linux/macOS).
- Interprets or JIT-compiles bytecode into native machine instructions.

# Java Compilation, JVM, Interpreter and JIT Notes

## Why is Java called both Compiled and Interpreted?

- **Compilation** → The Java compiler (`javac`) converts **`.java`** **source code** into platform-independent **bytecode (****`.class`****)**.
- **Execution** → The **JVM** executes the bytecode by interpreting it initially and **JIT-compiling** frequently used code into native machine code for better performance.

---
## What is Bytecode?
Bytecode is an intermediate code generated by the Java compiler.

Characteristics:
- Not human-readable
- Not machine code
- Platform-independent
- Executed by the JVM

Because Java generates bytecode instead of machine code directly, the same program can run on different operating systems.

"Write Once, Run Anywhere (WORA)"

---
### **JVM Execution**

- **Phase 1 – Parsing & Verification** → JVM reads the bytecode, verifies its correctness, organizes instructions, and prepares for execution.
- **Phase 2 – Execution** → Executes instructions sequentially (line-by-line) using the interpreter.

### **Interpreter**

- **Role** → Reads bytecode, executes one instruction at a time, then moves to the next (without pre-compiling the entire program).
- **Pros** → Fast startup, easy debugging, platform-independent.
- **Cons** → Slower for frequently executed code since the same instructions may be interpreted repeatedly.

---
- **Compiled** → `javac` compiles **`.java`** **source code** into **platform-independent bytecode (****`.class`****)**.
- **Interpreted + JIT Compiled** → The **JVM** executes the bytecode by interpreting it initially and **JIT-compiling** frequently executed (“hot”) code into native machine code for better performance.
---
# JVM = Interpreter + JIT Compiler

1. Interpreter
2. JIT Compiler
working together.

Interpreter:
- Quick startup
- Executes code immediately

JIT Compiler:
- Optimizes frequently used code
- Improves performance

This combination gives Java both **portability and speed.**

## 3. How Java Code Runs

```
Test.java  ──[javac]──►  Test.class (Bytecode)  ──[JVM]──►  Machine Code
(source)    Compilation    (platform-independent)  Interpretation  (platform-dependent)
```

1. **Source Code** is written in `.java` files.
2. **`javac` compiles** it into platform-independent **bytecode** (`.class` files).
3. **JVM interprets or JIT-compiles** the bytecode into native machine code for the specific OS/hardware.

> This is why Java is called "compiled + interpreted." The bytecode stage is what makes it portable.

## 6. Strings

### What is a String?
`String` is a **class** in Java (in `java.lang` package) — not a primitive. It represents a sequence of characters.

### String Creation
```java
String s1 = "Hello";              // String literal — stored in String Pool
String s2 = new String("Hello");  // Using constructor — stored on Heap
```

### String Pool vs Heap
```java
String str1 = "Hello";
String str2 = "Hello";
String str3 = new String("Hello");

System.out.println(str1 == str2);        // true  (same reference in pool)
System.out.println(str1 == str3);        // false (different objects)
System.out.println(str1.equals(str3));   // true  (same content)
```

> **Critical interview point:** Always use `.equals()` to compare String **content**. `==` compares **references** (memory addresses).

### String Immutability
Strings in Java are **immutable** — once created, their content cannot be changed. Every "modification" creates a new String object.

```java
String name = "John";
name.toUpperCase();               // Does NOT change name — creates new String
System.out.println(name);        // Still prints "John"

name = name.toUpperCase();        // Correct — reassign the reference
System.out.println(name);        // Now prints "JOHN"
```

> **Why immutable?** Thread safety, security (e.g., passwords, class loading), and String Pool caching all depend on immutability.

### Common String Methods
```java
String text = "Hello World";

text.length()           // 11
text.charAt(0)          // 'H'
text.substring(0, 5)    // "Hello"
text.contains("World")  // true
text.startsWith("He")   // true
text.endsWith("ld")     // true
text.replace("World", "Java")  // "Hello Java"
text.toUpperCase()      // "HELLO WORLD"
text.toLowerCase()      // "hello world"
text.trim()             // removes leading/trailing whitespace
text.split(" ")         // ["Hello", "World"]
text.indexOf("o")       // 4
```

### StringBuilder vs String
- `String` — immutable; bad for repeated concatenation in loops (creates many objects).
- `StringBuilder` — mutable, **not thread-safe (no synchronization).**, fast for single-threaded string building.
- `StringBuffer` — mutable, **thread-safe**, slower than StringBuilder  due to synchronization..

### Interview Points

- `StringBuilder` extends `AbstractStringBuilder`.
- Both `StringBuilder` and `StringBuffer` internally use a **resizable character array**.
- Default initial capacity = **16 characters**.
- If capacity is exceeded, Java automatically increases it (roughly **oldCapacity × 2 + 2**).
- `toString()` creates an immutable `String` from the builder.


> **Short-circuit evaluation:** `&&` stops at first `false`. `||` stops at first `true`. This matters for null checks: `if (obj != null && obj.getValue() > 0)` — safe because if obj is null, the second part never runs.

# Java Memory Management

#### What is Memory Management?
When a Java program runs, it needs memory to store things.

Stack  → Method calls and local variables
Heap   → Objects  - The actual object is in Heap. 
MetaSpace → Class information

Stack = temporary method memory - tack memory is short-lived.
Heap = object memory
MetaSpace = class memory

---
# Heap Memory

Managed by : **Garbage Collection (GC)** means cleaning unused objects from Heap memory.
## Heap Structure

```text
Heap
├── Young Generation
│   ├── Eden : New objects first go to Eden (**new object area**)
│   ├── Survivor 0 (S0 : Survivor space stores objects that survived Minor GC.
│   └── Survivor 1 (S1)
│
└── Old Generation (Tenured) - If an object survives many Minor GCs, JVM 
								assumes: This object is long-lived
```

### Object Lifecycle

1. **Created** → Object is created in **Eden** (new object area).
2. **Survives Minor GC** → Moves to a **Survivor** space. (An object **survives** if **at least one reference still points to it** when GC runs.)
3. **Survives multiple Minor GCs** → Promoted to the **Old Generation**.
4. **Becomes unreachable** → Eligible for Garbage Collection.
5. **Major/Full GC** → Removes unused objects from the Old Generation.

---
# Stack Memory

Each **thread gets its own stack**.
Each method call creates a **Stack Frame**.

A Stack Frame stores:
- Method parameters
- Local variables
- Object references
- Return address
- Operand stack

When the method returns, the entire frame is removed automatically.
> Stack memory is **not managed by GC**.
---
# Heap vs Stack

| Heap                         | Stack                               |
| ---------------------------- | ----------------------------------- |
| Stores objects               | Stores local variables & references |
| Shared                       | Per thread                          |
| GC managed                   | Auto removed after method returns   |
| Slower                       | Faster                              |
| Can throw `OutOfMemoryError` | Can throw `StackOverflowError`      |

---

## Types of Garbage Collection (GC)

- **Minor GC** → Cleans the **Young Generation** (Eden + Survivor). Fast and frequent.
- **Major GC** → Cleans the **Old Generation**. Slower and more expensive.
- **Full GC** → Cleans almost the **entire Heap**. Most expensive; causes longer application pauses.

> **Minor GC < Major GC < Full GC** (in terms of cost)

---

## `System.gc()`

```java
System.gc();
```

> It is only a **request**, not a command. The JVM may ignore it.

Stores **class-level information**, not objects.

Includes:

- Bytecode

- Static members

> Introduced in Java 8, replacing PermGen. Uses **native memory**, not Heap memory.

## String Pool

```java

```

Both references point to the **same String object** in the String Pool.

```java

```

---

## Common Errors

- **StackOverflowError** → Stack memory exhausted (usually due to infinite/deep recursion).
- **OutOfMemoryError** → Heap memory exhausted (too many objects, memory leaks, huge allocations).

## 11. OOP — The 6 Pillars

> **OOPS Definition:** A programming paradigm that uses objects and classes to design and implement software solutions.

![object_oriented_programming](https://media.geeksforgeeks.org/wp-content/uploads/20260605122433185064/object_oriented_programming.webp "Click to enlarge")
### 1. Class
A **blueprint** for creating objects. Defines the structure (fields) and behavior (methods) of objects.

```java
class Car {
    String color;   // field
    int speed;      // field

    void drive() { // method
        System.out.println("Car is driving");
    }
}
```

- ***State****: Represents the current data or attributes of an object.
- ***Behavior****: Represents the actions that an object can perform.
- ***Identity***: Every object has a unique identity in memory that distinguishes it from other objects, even if they contain the same data.
### 2. Object
An **instance** of a class. Represents a real-world entity with attributes (state) and behaviors (methods).

```java
Car myCar = new Car();   // creates object on heap
myCar.color = "Red";     // access field
myCar.drive();           // call method
```

> **Class vs Object:** Class is like the cookie cutter, object is the cookie.

### 3. Encapsulation
**Bundling data (fields) and methods into a single unit (class), and restricting direct access using access modifiers.**

The standard pattern: make fields `private`, expose them via `public` getters/setters.

```java
class Employee {
    private String name;    // hidden from outside
    private int salary;

    public String getName() { return name; }          // getter
    public void setName(String name) { this.name = name; }  // setter
}
```

> **Why encapsulation?** Controls what can be changed and how. Prevents invalid states. Makes code maintainable.

### 4. Inheritance
**A class (child/subclass) acquires properties and methods of another class (parent/superclass).** Promotes code reusability.

```java
class Animal {
    void eat() { System.out.println("This animal eats food."); }
}

class Dog extends Animal {   // Dog inherits eat() from Animal
    void bark() { System.out.println("Dog barks."); }
}
```

**Types of Inheritance in Java:**
- **Single** — one child from one parent (`Dog extends Animal`)
- **Multilevel** — chain: `C extends B extends A`
- **Hierarchical** — multiple children from one parent (`Dog extends Animal`, `Cat extends Animal`)

> ⚠️ **Java does NOT support Multiple Inheritance** (one class having two parents) — to avoid the Diamond Problem. Java solves this via **Interfaces**.

### 5. Polymorphism
**"Many forms"** — methods performing different tasks based on the object calling them.

#### Compile-Time Polymorphism (Method Overloading)
Same method name, **different parameter lists** — resolved at **compile time**.

```java
class Calculator {
    int add(int a, int b) { return a + b; }           // 2 params
    int add(int a, int b, int c) { return a + b + c; } // 3 params
}
// Overloading by: different number of params, different types, different order
```

#### Runtime Polymorphism (Method Overriding)
Subclass provides its own implementation of a parent class method — resolved at **runtime** based on the actual object type.

```java
class Animal { void sound() { System.out.println("Animal makes a sound"); } }
class Dog extends Animal { void sound() { System.out.println("Dog barks"); } }
class Cat extends Animal { void sound() { System.out.println("Cat meows"); } }

Animal a1 = new Dog();   // Upcasting
Animal a2 = new Cat();
a1.sound();   // "Dog barks"   — runtime decides which sound()
a2.sound();   // "Cat meows"
```

> **Key difference:** Overloading = same class, different signatures. Overriding = parent-child, same signature, different implementation.

### 6. Abstraction
**Showing only essential details, hiding the implementation.** Achieved through **abstract classes** and **interfaces**.

#### Abstract Class
```java
abstract class Animal {
    public abstract void sayHello();  // abstract — no body, MUST be overridden
    public abstract void sayBye();

    public void sleep() {             // concrete — has body, optionally overridden
        System.out.println("zzzzz...");
    }
}

class Dog extends Animal {
    public void sayHello() { System.out.println("Woof"); }
    public void sayBye()   { System.out.println("Woof Woof"); }
}
```

**Abstract Class Rules:**
- Declared with `abstract` keyword.
- Can have both abstract methods (no body) and concrete methods (with body).
- **Cannot be instantiated directly** (`new Animal()` → compile error).
- Acts as a blueprint — subclasses must implement all abstract methods (or themselves be abstract).

## 13. Interface

### What is an Interface?
- Defines a **contract** — what a class must do, not how.
- All methods are `public abstract` by default (Java 7 and before).
- Java 8+ allows **`default` methods** (methods with a body) inside interfaces.
- A class **implements** an interface (not extends).

```java
interface Drawable {

class Circle implements Drawable {

}
```

## 14. Multithreading

### Foundation Concepts (Build Up)

#### CPU
The brain of the computer — executes instructions, performs arithmetic, logic, control, and I/O operations. Modern CPUs have **multiple cores**.

#### Core
An individual processing unit within a CPU. A quad-core CPU can run 4 tasks truly simultaneously — one per core.

#### Program
A set of instructions written in a programming language. It's a static entity on disk (e.g., Microsoft Word `.exe`).

#### Process
An **instance of a program being executed**. When you open MS Word, the OS creates a process to manage its execution. A process has its own memory space.

#### Thread
The **smallest unit of execution within a process**. A process can have multiple threads that **share the same memory/resources** but run independently.

> Google Chrome example: each browser tab is a thread within the Chrome process.

---

### Multitasking vs Multithreading

| | Multitasking | Multithreading |
|---|---|---|
| **Level** | OS level — multiple **processes** | Application level — multiple **threads** within one process |
| **Managed by** | Operating System | JVM + OS |
| **Memory** | Each process has separate memory | Threads share same process memory |
| **Granularity** | Coarser | Finer |
| **Example** | Running Word + Firefox + music player | Word handling user input + spell checker simultaneously |

---

### Multithreading in Java
**Definition:** Concurrent execution of two or more threads within a single process to maximize CPU utilization.

Java's multithreading is part of the `java.lang` package — easy to implement.

#### Single-Core Environment
JVM + OS **rapidly switch between threads** (time-slicing) to give the **illusion of concurrency**. Not truly parallel.

#### Multi-Core Environment
JVM can **distribute threads across cores** — **true parallel execution** of threads.

### Thread Creation — Two Ways

#### Way 1: Extend `Thread` class
```java
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Thread running: " + Thread.currentThread().getName());
    }
}

// Usage:
MyThread t = new MyThread();
t.start();   // ← always call start(), NOT run() directly
```

#### Way 2: Implement `Runnable` interface (Preferred)
```java
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("Runnable thread: " + Thread.currentThread().getName());
    }
}

// Usage:
Thread t = new Thread(new MyRunnable());
t.start();

// Lambda shorthand (Java 8+):
Thread t2 = new Thread(() -> System.out.println("Lambda thread"));
t2.start();
```

> **Why Runnable is preferred:** Java doesn't support multiple class inheritance. If your class extends `Thread`, it can't extend anything else. `Runnable` is just an interface — your class is free to extend another class.

### Thread Lifecycle / States

```
NEW ──[start()]──► RUNNABLE ──► RUNNING ──[wait()/sleep()/I-O]──► BLOCKED/WAITING
                                    │                                      │
                                    │◄─────────────────────────────────────┘
                                    │
                               [finishes]
                                    ▼
                               TERMINATED
```

| State             | Description                                                  |
| ----------------- | ------------------------------------------------------------ |
| **NEW**           | Thread created but `start()` not yet called                  |
| **RUNNABLE**      | Ready to run, waiting for CPU time                           |
| **RUNNING**       | Currently executing on CPU                                   |
| **BLOCKED**       | Waiting to acquire a monitor lock (`synchronized` block)     |

### The Main Thread
When a Java program starts, **one thread begins immediately — the main thread**. It executes the `main()` method. All other threads are typically spawned from this thread.

### Synchronization (Key Interview Topic)

When multiple threads share data, **race conditions** can occur. `synchronized` prevents this by ensuring only one thread executes a block at a time.

```java
class Counter {
    private int count = 0;

    public synchronized void increment() {  // only one thread at a time
        count++;
    }

    public synchronized int getCount() {
        return count;
    }
}
```

> **`synchronized` keyword** puts a **monitor lock** on the object. Other threads trying to enter a synchronized method/block on the same object go to **BLOCKED** state.

---

### `wait()` and `notify()`

Used for inter-thread communication (inside `synchronized` blocks).

```java
// Thread A: producer
synchronized(lock) {
    // produce data
    lock.notify();   // wake up a waiting thread
}

// Thread B: consumer
synchronized(lock) {
    lock.wait();     // release lock and wait to be notified
    // consume data
}
```

| Method | What it does |
|---|---|
| `wait()` | Releases the lock, thread goes to WAITING state |
| `notify()` | Wakes up one thread waiting on this object |
| `notifyAll()` | Wakes up all threads waiting on this object |

> These must always be called **inside a `synchronized` block**, otherwise `IllegalMonitorStateException` is thrown.

# Stream API 

## What is a Stream?
A Stream is a sequence of elements used for **processing data**, not storing it.

```
Collection
     ↓
   Stream
     ↓
Intermediate Operations
     ↓
Terminal Operation
     ↓
   Result
```

### Properties
- Doesn't modify original collection
- Supports functional programmin
- Lazy by default
- Can be processed sequentially or in parallel
- Cannot be reused after a terminal operation

###  **Why Stream API?**  
The Stream API (introduced in Java 8) was created to make processing collections of data faster, cleaner, and more readable. Think of it as a pipeline that data flows through to get filtered, transformed, and aggregated.  
**The Core Benefits:**  

- **Declarative Code (What, not How):** Instead of writing complex loops and `if` conditions, you describe _what_ you want to do with the data.  
- **No Side Effects:** Streams do not modify the original data source (like a List); they return a new result.  
- **Easy Parallelism:** You can change `.stream()` to `.parallelStream()` to split the work across multiple CPU cores automatically.

## Autoboxing & Unboxing

Java Collections work only with **objects**, not primitive data types.

```java
List<Integer> list = new ArrayList<>();
```

Not:

```java
List<int> list = new ArrayList<>(); // ❌ Invalid
```

To make primitives work with objects, Java automatically converts between them.

---
### Autoboxing
**Autoboxing** is the automatic conversion of a **primitive** into its **wrapper object**.

```java
int x = 10;
Integer num = x;   // int → Integer
```

Java automatically performs:

```java
Integer num = Integer.valueOf(x);
```

---
### Unboxing

**Unboxing** is the automatic conversion of a **wrapper object** into its **primitive**.

```java
Integer num = 10;
int x = num;       // Integer → int
```

Java automatically performs:

```java
int x = num.intValue();
```

---
### Why is it needed?
- Collections store only **objects**, not primitives.
- Generics (`List<T>`) work only with objects.
- Makes code shorter by avoiding manual conversions.

---
### Common Pitfall ⚠️

```java
Integer num = null;
int x = num;   // NullPointerException
```

During unboxing, Java tries to call `num.intValue()`. Since `num` is `null`, it throws a `NullPointerException`.

---

### One-Line Revision

```
Autoboxing : Primitive → Wrapper Object
Unboxing   : Wrapper Object → Primitive
```

## 🎯 Interview Quick-Fire Answers

**Q: Why is String immutable in Java?**
A: Security (class loading, passwords), thread safety (can be safely shared), and String Pool caching (same literal reused by multiple variables).

**Q: `==` vs `.equals()` in Java?**
A: `==` compares references (memory address). `.equals()` compares content. For Strings, always use `.equals()`.

**Q: What is the difference between `abstract class` and `interface`?**
A: Abstract class can have constructors, instance fields, and concrete methods — use for IS-A with shared state. Interface defines a contract, supports multiple implementation — use for capabilities/behaviors.

**Q: What is method overloading vs overriding?**
A: Overloading = same name, different params, same class, resolved at compile time. Overriding = same name + params, parent-child classes, resolved at runtime (dynamic dispatch).

**Q: Why doesn't Java support multiple inheritance?**
A: Diamond problem — ambiguity when two parents have the same method. Java solves this through interfaces (multiple interface implementation is allowed).

**Q: `start()` vs `run()` in threads?**
A: `start()` creates a new thread and calls `run()` in that new thread. Calling `run()` directly just executes it in the **current thread** — no new thread is created.

**Q: What is the String Pool?**
A: A special area in the heap (PermGen in Java 7, Heap in Java 8+) where string literals are interned. When you write `"Hello"`, JVM checks the pool first — if it exists, reuses the same reference. This is why `str1 == str2` is `true` for literals.

**Q: `ArrayList` vs `LinkedList`?**
A: ArrayList = dynamic array, O(1) random access, O(n) insert/delete in middle. LinkedList = doubly-linked list, O(n) access, O(1) insert/delete if you have the node. ArrayList is preferred for most use cases due to cache locality.

**Q: `HashMap` vs `Hashtable`?**
A: Hashtable is legacy, synchronized (thread-safe but slow), doesn't allow null keys/values. HashMap is not synchronized (use ConcurrentHashMap for thread safety), allows one null key and multiple null values.

**Q: What are the thread states in Java?**
A: NEW → RUNNABLE → RUNNING → (BLOCKED / WAITING / TIMED_WAITING) → TERMINATED.

- `Collection` stores individual elements, whereas `Map` stores **key-value pairs**. Since its data model is different, `Map` was designed as a separate interface and does not extend `Collection`.
-  **Collection vs Collections**
	- **Collection** is an interface representing a group of objects.
	- **Collections** is a utility class containing static helper methods like `sort()`, `reverse()`, and `binarySearch()`.
	- **Collection = data structure**, **Collections = helper methods**.

### **Java HashMap & Concurrency Interview Guide**

**Q: How does HashMap work internally?** 
A: `HashMap` stores data as **key-value pairs** in an array of buckets. The key’s `hashCode()` determines the bucket index. If multiple keys map to the same bucket (collision), they’re stored using a linked list or a Red-Black Tree (Java 8+). Lookup computes the bucket, then uses `equals()` to find the correct key.

**Q: What is hashing?** 
A: Hashing converts an object into an integer hash value using `hashCode()`. This hash helps determine where the key should be stored, enabling average O(1) insertion and lookup.

**Q: What is hashCode()?** 
A: `hashCode()` returns an integer representing an object’s hash value. Hash-based collections like `HashMap` and `HashSet` use it to determine the bucket where an object should be stored.

**Q: What is equals()?** 
A: `equals()` checks **logical equality** between two objects, whereas `==` checks if both references point to the same object. HashMap uses `equals()` to identify the correct key within a bucket.

**Q: Why override equals() and hashCode() together?** 
A: If two objects are equal according to `equals()`, they **must** return the same `hashCode()`. Otherwise, collections like `HashMap` may fail to locate stored objects correctly.

**Q: What is a collision?** 
A: A collision occurs when two different keys produce the same bucket index. HashMap resolves collisions using linked lists or Red-Black Trees (Java 8+).

**Q: What is chaining?** 
A: Chaining is the collision-resolution technique where multiple entries sharing the same bucket are linked together. Before Java 8 this was a linked list; Java 8 may convert long chains into Red-Black Trees.

**Q: What is the load factor?** 
A: The load factor determines when HashMap should resize. Default value is **0.75**, balancing memory usage and performance.

**Q: What is the initial capacity?** 
A: The default initial capacity is **16** buckets. It is always maintained as a power of two for efficient index calculation.

**Q: What is rehashing?** 
A: Rehashing occurs after resizing. A larger bucket array is created, and all existing entries are redistributed based on their hash values.

**Q: What is resizing?** 
A: When the number of entries exceeds **capacity × load factor**, HashMap doubles its capacity and rehashes all entries.

**Q: Why is HashMap O(1)?** 
A: Because hashing allows direct access to the target bucket. With a good hash function and low collisions, lookup, insert, and delete take constant time on average.

**Q: When does HashMap become O(n)?** 
A: If many keys collide into the same bucket (especially before Java 8), operations degrade to O(n) because the linked list must be traversed. Java 8 improves this to O(log n) by using Red-Black Trees for large buckets.

**Q: Why were Red-Black Trees introduced in Java 8?** 
A: To improve worst-case performance during heavy collisions. Large linked lists are converted into Red-Black Trees, reducing lookup time from O(n) to O(log n).

**Q: HashMap vs Hashtable** 
A: `HashMap` is **not thread-safe**; `Hashtable` is thread-safe. `HashMap` allows one null key and multiple null values; `Hashtable` allows neither. `HashMap` is generally faster because it avoids synchronising every operation.

**Q: Hashtable vs ConcurrentHashMap** 
A: `Hashtable` synchronizes every method, causing high contention. `ConcurrentHashMap` uses finer-grained synchronization/CAS, allowing multiple threads to operate concurrently and improving scalability.

**Q: Why isn’t HashMap thread-safe?** 
A: Multiple threads can modify the internal bucket structure simultaneously, leading to race conditions, inconsistent data, or even corruption during resize operations.

**Q: What problems occur with concurrent writes?** 
A: Concurrent writes can cause lost updates, inconsistent reads, corrupted bucket structures, and unpredictable behavior because HashMap performs no synchronization.

**Q: How does ConcurrentHashMap improve performance?** 
A: It minimizes locking by allowing concurrent reads and fine-grained updates (using bucket-level locking/CAS). Multiple threads can operate on different buckets simultaneously.

**Q: Synchronization vs Lock Striping** 
A: Synchronization locks the entire collection for each operation. Lock striping (or fine-grained locking) locks only a portion of the data, allowing higher concurrency and better performance.

**Q: Can ConcurrentHashMap store null?** 
A: No. It disallows null keys and null values to avoid ambiguity in concurrent operations (e.g., distinguishing “key absent” from “mapped to null”).

**Q: Iterator vs ListIterator** 
A: `Iterator` works for all collections and supports only forward traversal. `ListIterator` works only with `List`, supports both forward and backward traversal, and allows element modification during iteration.

Q: How does HashMap calculate the bucket index? 
A: It uses a bitwise AND operation: `index = (n - 1) & hash` (where `n` is capacity). Because capacity is a power of 2, `(n - 1)` acts as a bitmask. This is vastly faster for the CPU than using the modulo operator (`%`).  

Q: When does a Linked List turn into a Red-Black Tree? 
A: When a single bucket hits 8 elements (`TREEIFY_THRESHOLD`) AND the total map capacity is at least 64. If the total capacity is under 64, Java will just resize the map instead of treeifying.  

Q: When does a Red-Black Tree turn back into a Linked List? 
A: During a resize operation, if a tree bucket's element count drops to 6 or fewer (`UNTREEIFY_THRESHOLD`), Java converts it back to a linked list to save memory overhead.  

Q: How does HashMap handle a null key? 
A: It bypasses the normal hashing and forces the hash value to be 0. Because of this, a `null` key is always stored in the very first slot (bucket index 0).  

Q: What makes an ideal HashMap key? 
A: An immutable object (like `String` or `Integer`). If a key's fields change after insertion, its `hashCode()` changes. The map will look in the wrong bucket, making the object unretrievable and causing a silent memory leak. 

Q: What is the Java 7 "Infinite Loop" resize problem? 
A: In Java 7, concurrent threads resizing a map could reverse the linked list order simultaneously. This created a circular reference (A points to B, B points to A), trapping the `get()` method in an infinite loop that pegged the CPU at 100%. ***Java 8 fixed this by preserving insertion order during resizes.***

**Q: HashSet vs LinkedHashSet**  
**A:** `HashSet` stores unique elements with no ordering. `LinkedHashSet` maintains insertion order using a linked list internally, with slightly higher memory overhead.

**Q: TreeSet vs HashSet**  
**A:** `HashSet` provides average O(1) operations with no ordering. `TreeSet` stores elements in sorted order using a Red-Black Tree, with O(log n) operations.

**Q: Why are duplicates not allowed in Set?**  
**A:** A Set enforces uniqueness by design. Before inserting an element, it checks whether an equivalent element already exists.

**Q: How does HashSet detect duplicates?**  
**A:** `HashSet` internally uses a `HashMap`. It uses `hashCode()` to locate the bucket and `equals()` to verify if the element already exists.

**Q: How does HashSet work internally?**  
**A:** `HashSet` is backed by a `HashMap`. Elements are stored as keys, while a dummy object is used as the value. Hence, duplicate keys are automatically rejected.

**Q: Queue vs Deque**  
**A:** `Queue` primarily supports FIFO operations. `Deque` (Double Ended Queue) allows insertion and deletion from both the front and rear.

**Q: ArrayDeque vs Stack**  
**A:** `Stack` is a legacy synchronized class. `ArrayDeque` is faster, unsynchronized, and recommended for implementing stacks in modern Java.

**Q: How does PriorityQueue work internally?**  
**A:** `PriorityQueue` is implemented using a Binary Heap. The smallest (or highest-priority) element is always kept at the root.

**Q: What is a Heap?**  
**A:** A Heap is a complete binary tree that maintains the heap property. Java’s `PriorityQueue` uses a Min Heap by default.

**Q: FIFO Queue vs PriorityQueue**  
**A:** A normal Queue removes elements in insertion order (FIFO). A `PriorityQueue` removes elements based on priority, regardless of insertion order.

## **TreeMap / TreeSet 

**Q: Why are elements automatically sorted?**  
**A:** Both `TreeMap` and `TreeSet` maintain elements in sorted order using a Red-Black Tree and compare elements during insertion.

**Q: Comparable vs Comparator**  
**A:** `Comparable` defines the class’s natural ordering. `Comparator` defines custom ordering externally without modifying the class.

**Q: What is a Red-Black Tree?**  
**A:** A self-balancing Binary Search Tree that guarantees O(log n) search, insertion, and deletion by maintaining balanced height.

**Q: What is natural ordering?**  
**A:** The default sorting order defined by implementing the `Comparable` interface (e.g., ascending integers or alphabetical strings).

**Q: What is custom sorting?**  
**A:** Sorting based on user-defined logic using a `Comparator`, such as sorting students by age or salary.

---
##  **Comparable vs Comparator 

**Q: Comparable vs Comparator**  
**A:** `Comparable` provides one default (natural) ordering inside the class. `Comparator` provides custom ordering outside the class and supports multiple sorting strategies.

**Q: Which one modifies the class?**  
**A:** **Comparable = sorting logic goes INSIDE the class.**  **(modifies the class)**
 **Comparator = sorting logic stays OUTSIDE the class.** 

**Q: Which one supports multiple sorting criteria?**  
**A:** `Comparator`. You can create multiple comparators (e.g., sort by name, age, salary) without changing the class.

**Q: What is an Anonymous Comparator?**  
**A:** A comparator created using an anonymous inner class, useful for one-time custom sorting.

**Q: What is a Lambda Comparator?**  
**A:** A concise Java 8+ way to implement a `Comparator` using lambda expressions, improving readability.

# **Generics 

**Q: Why were Generics introduced?**  
**A:** To provide **compile-time type safety**, eliminate explicit casting, and catch type-related errors early.

**Q: What is a raw type?**  
**A:** A generic class used **without specifying a type** (e.g., `List list`). It bypasses type checking and is generally discouraged.

**Q: What is type safety?**  
**A:** Ensures only the specified type can be stored in a generic collection, preventing `ClassCastException` at runtime.

**Q: What are Generics?**  
**A:** Generics allow classes, interfaces, and methods to work with different data types while maintaining type safety.

**Q: What is the** **`?`** **wildcard?**  
**A:** Represents an **unknown type** when the exact generic type is not important.

**Q: What is** **`? extends T`****?
**A:** Accepts **T or its subclasses**. Used mainly for **reading** data (Producer).

**Q: What is** **`? super T`****?**  
**A:** Accepts **T or its superclasses**. Used mainly for **writing** data (Consumer).

**Q: When should you use** **`extends`** **vs** **`super`****?**  
**A:** **PECS:** _Producer Extends, Consumer Super._

**Q: What is type erasure?** _(Good follow-up)_  
**A:** Generic type information is removed at compile time, so generics exist only during compilation.

# **Exception Handling 

**Q: What is an Exception?**  
**A:** An event that disrupts the normal flow of program execution and can be handled to prevent program termination.

**Q: Checked vs Unchecked Exception**  
**A:** Checked exceptions are checked at compile time and must be handled or declared. Unchecked exceptions occur at runtime and are optional to handle.

**Q:** **`throw`** **vs** **`throws`**  
**A:** `throw` explicitly throws an exception. `throws` declares that a method may throw exceptions.

**Q: What is** **`finally`****?
**A:** A block that executes regardless of whether an exception occurs, typically used for cleanup.

**Q: Can** **`finally`** **not execute?** _(Common follow-up)_  
**A:** Yes, in rare cases such as `System.exit()` or JVM crash.

**Q: What is try-with-resources?**  
**A:** Automatically closes resources (e.g., files, streams) after use, eliminating the need for explicit cleanup.

**Q: Why use try-with-resources?**  
**A:** Prevents resource leaks and produces cleaner code.

**Q: What is a custom exception?**  
**A:** A user-defined exception created by extending `Exception` or `RuntimeException` to represent application-specific errors.

**Q: Exception vs Error** _(Very common)_  
**A:** Exceptions can usually be handled by applications. Errors represent serious JVM/system failures and are generally not recoverable.

**Q: Can we have multiple** **`catch`** **blocks?**  
**A:** Yes. They should be ordered from the most specific exception to the most general.

**Q: Can we have** **`try`** **without** **`catch`****?
**A:** Yes, if it is followed by a `finally` block or uses try-with-resources.
